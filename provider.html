El problema real
¿Qué pasa cuando todo vive en la misma clase?
El problema — todo mezclado en un solo widget
class MapScreen extends StatefulWidget { // GPS directo aqui late StreamSubscription _gpsSub; // HTTP directo aqui Future<void> _cargarMunicipios() async { final resp = await http.get(...); final data = jsonDecode(resp.body); // ... 50 lineas de parseo } // SQLite directo aqui Future<void> _guardar(Colecta c) async { final db = await openDatabase(...); await db.insert('colectas', c.toMap()); } // Y tambien toda la UI aqui Widget build(...) { /* 200 lineas */ } }
⚠️
El síntoma
Un archivo de 400 líneas donde está todo mezclado: la UI, el GPS, el HTTP y la base de datos. Si cambias SQLite por otro storage, tienes que tocar el widget.
💡
La solución
Separar en 3 capas. Cada archivo sabe solo lo que le toca. La UI solo construye widgets.
La arquitectura
3 capas — una regla que lo explica todo
📱
Capa UI — lib/screens/ + lib/widgets/
Solo construye pantallas. Nunca toca SQLite ni HTTP directamente.

La regla de oro
Las flechas solo bajan. UI llama a Services. Services llama a Data. Nunca al revés. Si un archivo de Data importa un widget, algo está mal.

// Solo datos: campos y conversion Map class Colecta { final int id; final String tipo; final double latitud; final double longitud; final int timestamp; final bool sincronizado; // Sin widgets, sin sqflite imports Map<String,dynamic> toMap() => { 'id': id, 'tipo': tipo, 'latitud': latitud, 'longitud': longitud, 'timestamp': timestamp, 'sincronizado': sincronizado ? 1 : 0, }; factory Colecta.fromMap(Map m) => Colecta( id: m['id'], tipo: m['tipo'], latitud: m['latitud'], longitud: m['longitud'], timestamp: m['timestamp'], sincronizado: m['sincronizado'] == 1, ); }

Capa Services
GpsService — el GPS desacoplado de la pantalla
lib/services/gps_service.dart
// Sin imports de Flutter UI // Sin referencias a BuildContext import 'package:geolocator/geolocator.dart'; import 'package:latlong2/latlong2.dart'; class GpsService { StreamSubscription<Position>? _sub; final _ctrl = StreamController<LatLng>.broadcast(); // El Screen escucha este stream Stream<LatLng> get posicion => _ctrl.stream; Future<void> iniciar() async { final ok = await _pedirPermisos(); if (!ok) return; _sub = Geolocator .getPositionStream( locationSettings: LocationSettings( distanceFilter: 5, )) .listen((pos) { _ctrl.add(LatLng( pos.latitude, pos.longitude)); }); } // Haversine vive aqui, no en la UI double distancia( LatLng a, LatLng b) { /* ... */ } void detener() { _sub?.cancel(); _ctrl.close(); } }
✓
Por qué GpsService mejora el código
MapScreen ya no gestiona StreamSubscription ni permisos. Solo escucha el stream. Si cambias geolocator por otra librería, solo tocas GpsService.
💡
broadcast()
StreamController.broadcast() permite que múltiples widgets escuchen el mismo stream GPS simultáneamente — el mapa, un contador de distancia, y un panel de estado pueden actualizarse con una sola fuente.

MapScreen — limpia, sin lógica de negocio

class _MapScreenState extends State<MapScreen> { // Instancia del servicio final _gps = GpsService(); final _ctrl = MapController(); final _trayectoria = <LatLng>[]; @override void initState() { super.initState(); // El Screen solo llama al Service _gps.iniciar(); } @override void dispose() { _gps.detener(); // limpieza super.dispose(); } @override Widget build(BuildContext ctx) => StreamBuilder<LatLng>( stream: _gps.posicion, builder: (ctx, snap) { if (snap.hasData) { _trayectoria.add(snap.data!); _ctrl.move( snap.data!, _ctrl.camera.zoom); } return FlutterMap( mapController: _ctrl, children: [ TileLayer(...), PolylineLayer(...) ], ); }, ); }
✓
MapScreen sin lógica
No hay Geolocator.getPositionStream(). No hay manejo de permisos. No hay Haversine. Solo escucha el stream y actualiza la UI. Eso es lo que debe hacer un widget.

La misma lógica de GpsService puede usarse desde CollectScreen para obtener la posición al abrir el formulario — sin duplicar código.

IA como copiloto
Prompt por capa — la IA genera código limpio
1
Primero el modelo
"Genera la clase Dart Colecta con campos id, tipo, latitud, longitud, timestamp y sincronizado. Con toMap() y fromMap() para sqflite. Sin imports de Flutter."
2
Luego la capa Data
"Genera ColectaDb en Dart con sqflite. Operaciones: crear tabla, insertar Colecta, listar todas, listar pendientes de sincronizar. Sin widgets."
3
Luego la capa Services
"Genera GpsService en Dart puro que expone un Stream de LatLng usando geolocator con distanceFilter: 5. Sin BuildContext. Con método iniciar() y detener()."
4
Finalmente el Screen
"Genera MapScreen Flutter que usa GpsService para obtener posición GPS y dibuja la trayectoria con PolylineLayer de flutter_map. Sin lógica de GPS directa en el widget."
💡
Pedir cada capa por separado produce código más limpio que pedir toda la app de una vez. La IA respeta los límites cuando el prompt los declara explícitamente.
Actividad práctica
Refactoriza: mueve el GPS de MapScreen a GpsService
1
Crea lib/services/gps_service.dart
Mueve todo el código de Geolocator.getPositionStream() al nuevo archivo. El Screen no debe importar geolocator.
2
Crea lib/data/colecta_db.dart
Mueve las funciones openDatabase(), insert() y query() del formulario al nuevo archivo. Solo devuelve List<Colecta> o int.
3
Simplifica MapScreen
MapScreen solo instancia GpsService, llama a iniciar() en initState() y detener() en dispose(). La lógica GPS ya no existe en el widget.
4
Verifica la regla de oro
Revisa cada import. ¿Hay algún import de sqflite en map_screen.dart? ¿Algún import de flutter/material en colecta_db.dart? Si lo hay, la capa está mal.
Resumen
Lo que aprendiste hoy
  • 3 capas: UI (screens/widgets), Services (lógica), Data (SQLite/HTTP) — cada archivo sabe solo lo que le toca
  • La regla de oro: las flechas solo bajan. Data nunca importa Services. Services nunca importa widgets.
  • GpsService expone un Stream que los Screens escuchan — el GPS vive fuera de la UI
  • ColectaDb devuelve List<Colecta> y int — nunca widgets ni contexto Flutter
  • main.dart solo tiene runApp() — sin frameworks de estado externos
  • La IA genera código más limpio cuando el prompt especifica la capa: "sin widgets", "sin BuildContext", "sin lógica de negocio"
